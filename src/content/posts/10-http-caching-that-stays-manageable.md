缓存这件事，刚接触时大家都很兴奋。

听起来它几乎只带来好处：更快、更省、更稳。

但项目做久了你会发现，缓存真正容易让人头疼的地方，不是配置不会写，而是配置写完之后，整个团队越来越没人能说清楚系统到底是怎么工作的。

用户看到旧页面，是浏览器缓存了，还是 CDN 没失效？
JS 更新了但页面没变，是 HTML 还在缓存，还是构建产物没有换 hash？
本地一切正常，线上偶发异常，到底是回源慢，还是根本没有回源？

这类问题一旦出现，说明缓存已经从“性能优化”变成“理解负担”了。

## 缓存最容易错的地方，是一把梭

很多团队第一次配缓存，喜欢统一处理。

“反正都缓存久一点，肯定更快。”

这句话只对了一半。静态资源、HTML、接口响应，本来就不是一回事。你把它们都按同一种逻辑缓存，后面几乎一定会出问题。

最基本的区分应该是：

- 带 hash 的静态资源，可以大胆长缓存
- HTML 页面要更谨慎，因为它承接的是内容更新
- API 缓存要看数据是否和用户、时间或状态强相关

这不是学院派讲究，而是实际维护成本问题。资源类型不分，后面排障会非常痛苦。

## 真正难的是失效，不是缓存本身

很多方案在“命中时”都很好看，真正见高下的是“要更新时”。

缓存最容易把团队拖进坑里的，就是失效机制不清楚。内容改了之后多久生效，谁负责主动刷新，哪些层会继续保留旧版本，团队是不是有一套一致预期，这些问题如果没有答案，缓存策略就只是“暂时看起来更快”。

我一般会特别在意这句话：

如果今天我要更新一个高流量页面，团队能不能说清楚新版本多久能稳定到所有用户手里？

说不清，后面一定会出事。只是时间问题。

## 所以我更喜欢可解释的缓存

不一定要最激进，但一定要可解释。

也就是说，团队最好能回答：

- 哪些资源走长缓存
- 哪些内容只做短缓存或验证缓存
- 出现旧内容时先查哪一层
- 发布后如果需要强制更新，流程是什么

缓存配得太“聪明”，但没人能维护，通常还不如保守一点。

因为前端项目真正可怕的，不是偶尔慢一点，而是线上出现“不知道为什么有人看到旧东西”的那种问题。它会直接吃掉团队的信心和时间。

缓存当然要做，但它的目标不该只是更快，而是又快又能被团队理解。
